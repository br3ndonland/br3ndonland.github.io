## Description

<a href="https://git-scm.com/" rel="external" target="_blank">Git</a> and <a href="https://code.visualstudio.com/" rel="external" target="_blank">Visual Studio Code</a> ("VSCode") are essential tools for software development. The built-in VSCode Git tools leave a lot to be desired, so VSCode <a href="https://code.visualstudio.com/docs/configure/extensions/extension-marketplace" rel="external" target="_blank">extensions</a> have been developed to support additional Git features. I used the <a href="https://github.com/gitkraken/vscode-gitlens" rel="external" target="_blank">GitLens</a> extension for years, and even <a href="https://github.com/gitkraken/vscode-gitlens/pull/1316" rel="external" target="_blank">contributed a feature to it</a>. The extension was later <a href="https://www.gitkraken.com/blog/gitkraken-acquires-gitlens-for-visual-studio-code" rel="external" target="_blank">acquired by GitKraken</a>. Over time, it became bloated with enormous numbers of settings, overcomplicated panels with graphs and dashboards, obtrusive paid features, and AI integrations. It also began regularly invalidating extension versions, which would disable the extension and display obnoxious notifications forcing users to update.

I eventually got fed up with GitLens and decided to write my own extension from scratch. Introducing <a href="https://github.com/br3ndonland/gitless-vscode" rel="external" target="_blank">GitLess</a>, a VSCode extension that offers the essential features of GitLens with less bloat. It is available on the <a href="https://open-vsx.org/extension/br3ndonland/gitless-vscode" rel="external" target="_blank">Open VSX registry</a> and <a href="https://marketplace.visualstudio.com/items?itemName=br3ndonland.gitless-vscode" rel="external" target="_blank">Microsoft marketplace</a>.

What GitLess _does not_ do:

- GitLess does not collect telemetry.
- GitLess does not expire extension versions or disable functionality until you update.
- GitLess does not require a separate account, subscription, or cloud service.
- GitLess does not turn your Source Control panel into a full workflow dashboard.
- GitLess does not run heavyweight background services just to show Git history.

## Development

I built the initial <a href="https://github.com/br3ndonland/gitless-vscode/releases/tag/0.1.0" rel="external" target="_blank">version 0.1.0</a> of this extension with the <a href="https://kilo.ai/" rel="external" target="_blank">Kilo Code</a> CLI (based on <a href="https://opencode.ai/" rel="external" target="_blank">OpenCode</a>) and their <a href="https://kilo.ai/docs/code-with-ai/agents/auto-model" rel="external" target="_blank">Auto Frontier</a> models, which were mostly Claude Opus 4.6 at the time. The initial turn took about 15 minutes in the (now deprecated) "orchestrator" mode and costed $3.15 in credits. As usual for code generated by Claude Opus models, the code was incoherent and nonfunctional. The extension did not start, "settings" did nothing, and "tests" were little more than placeholders. Anecdotally, I've found that Claude models often ignore their <a href="https://kilo.ai/docs/customize/custom-instructions" rel="external" target="_blank">instructions</a>, so I had to interrupt turns and steer the model when it started doing things I didn't want it to do. It was also relatively expensive because <a href="https://openrouter.ai/compare/anthropic/claude-opus-4.6/openai/gpt-5.4" rel="external" target="_blank">inference with Claude Opus 4.6 is twice as expensive per input token as inference with GPT-5.4</a>. My experience with Claude models in Claude Code has been largely the same, so I think this is more of a Claude model problem than a Kilo Code problem. The rest of the work on 0.1.0 took about 15 hours and costed about $30 in credits. So the initial turn only accomplished a small percentage of the work, and then it still needed a lot more work before it was functional. This is a typical experience in <a href="https://en.wikipedia.org/wiki/Prompt_engineering" rel="external" target="_blank">prompt engineering</a>. Prompting is an iterative process.

After burning through some more Kilo Code credits (about $100 in total) fixing bugs with Kilo Code and Claude Opus 4.6, I switched to <a href="https://developers.openai.com/codex" rel="external" target="_blank">Codex</a> with GPT-5.4. Development went more smoothly -- bugs were fixed, features actually worked, and tests evaluated real-world extension behavior. Codex paid attention to my <a href="https://developers.openai.com/codex/guides/agents-md" rel="external" target="_blank">instructions</a>, and I appreciated its relatively concise communication style. It was also cheaper. I was able to get a lot of work done on the $20/month Plus plan, especially if I paced myself to stay under the usage limits.

See my [dotfiles](/projects/dotfiles) repo for more details on the configurations I use for Codex, Kilo Code, and other AI coding tools.
